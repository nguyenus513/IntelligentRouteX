# IntelligentRouteX System Overview

## Purpose

IntelligentRouteX (IRX) is a Java-based grouped-delivery dispatch optimization system. It is designed as a complete thesis/demo platform: optimizer, API runtime, BigData-lite ingestion, Adaptive ML policy, Playground UI, event/artifact observability, and one-click local execution.

- Playground route `/playground` passes static, live, rescue, BigData, event, artifact, Adaptive ML, baseline, result, and raw JSON checks.

## What runs in the system

### Backend

- Java backend on `http://localhost:18116` by default.
- API base: `http://localhost:18116/api/v1`.
- Runtime store, queue routing, async worker MVP, event log, artifact store, metrics, idempotency, and access guards.

### Dashboard / Playground

- Dashboard dev server on `http://localhost:5173` by default.
- Main demo route: `http://localhost:5173/playground`.
- The Playground is API-first and consumes locked `/api/v1` contracts.

### External solvers

- VROOM can run through `VROOM_BASE_URL` or `VROOM_BIN`.
- The certified local setup uses WSL wrapper `tools/vroom/vroom-wsl.cmd` when available.
- PyVRP is used when Python runtime and package are available.
- If a solver is not configured, it must report `EVIDENCE_GAP` rather than producing unverified claims.

## Core dispatch modes

- Static dispatch: batch/static order-driver optimization.
- Live rolling: buffered dynamic orders, driver telemetry, rolling dispatch cycles, event output.
- Rescue: late-risk or disrupted route recovery with dominance rollback guard.
- BigData-lite: large input ingestion through batch APIs with async queue and paginated output.

## Optimization pipeline

1. Normalize input.
2. Build seed contributors.
3. Store candidates in `EliteSolutionArchive`.
4. Evaluate schedule, distance, late, capacity, coverage, and pickup/dropoff feasibility.
5. Run hybrid improvement and Adaptive ML policy.
6. Apply dominance and late-not-worse guards.
7. Emit final `IRX_ML_FUSED_HYBRID` result, artifacts, and events.

## Adaptive ML role

Adaptive ML is a policy layer, not a replacement solver. It participates in:

- Adaptive seed policy diagnostics.
- Adaptive operator policy.
- Adaptive move priority and move ordering.
- TOP-K assisted evaluation.
- Quality-seeking mode with expanded budget and deeper exact reorder.
- Reward tracking and diagnostics.

The evaluator, hard validators, and dominance guard remain the final authority.

## Claim boundaries

IRX can claim:

- API-first production-demo Java dispatch system.
- Hybrid optimization with external seed absorption and dominance guard.
- Adaptive ML policy in the optimization core for move ordering, TOP-K assisted evaluation, and quality-seeking search.
- BigData-lite runtime for large input ingestion without Kafka/Spark.
- One-screen Playground and one-click local launcher.

IRX should not claim:

- ML fully replaces VROOM/PyVRP/OR-Tools.
- BigData-lite is a distributed Kafka/Spark production cluster.
- FAST synthetic distances are production road benchmark claims.
- Native binary packaging is complete unless a separate EXE/installer milestone is added.
