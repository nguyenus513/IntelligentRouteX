# Adaptive Bundle Dispatch Live Rolling Benchmark

## Dataset

Synthetic deterministic live-style dispatch suite designed to prove Adaptive ML-Bundle Dispatch value on dispatch KPIs rather than only solver distance.

| Scenario | Orders | Drivers | Purpose |
|---|---:|---:|---|
| Normal Streaming | 600 | 60 | Continuous order arrival, convenient insertion proof |
| Rush-hour Burst | 900 | 70 | Burst load, waiting/late robustness |
| Dense City | 100 | 20 | High pickup density, same-zone dropoffs, bundle-rate proof |
| Sparse Orders | 100 | 20 | Far orders, guard against bad bundling |
| Driver Delay | 160 | 25 | 10% delayed drivers, break-risk and repair proof |
| Tight Deadlines | 180 | 25 | SLA pressure proof |
| Capacity Pressure | 220 | 28 | bundle/load pressure proof |
| Stress Test | 1000 | 100 | Runtime/scalability proof |

## Protocol

Compare live rolling dispatch baselines on identical generated data:

- `greedy_nearest_driver`
- `solver_only_rolling`
- `ortools_rolling`
- `vroom_rolling`
- `pyvrp_rolling`
- Ablations `A1_aging_regret` through `A5_destroy_repair`
- Full system `A6_full_adaptive`

Run:

```powershell
py -3 scripts/run_adaptive_bundle_dispatch_benchmark.py --output-dir artifacts/benchmark/adaptive_bundle_dispatch_benchmark
```

## Metrics

- Dispatch: `servedOrders`, `unassignedOrders`, `bundleCount`, `bundleRate`
- Route: `distance`, `avgStopsPerDriver`, `detourRatio`
- SLA: `lateCount`, `avgLateness`, `p95Lateness`, `avgWaitingTime`, `p95WaitingTime`, `maxOrderAge`
- Driver: `utilization`, `avgLoad`, `idleTime`, `routeChurn`, `runtimeP95Ms`
- New Layer: `convenientInsertions`, `breakRiskTriggered`, `destroyRepairTriggered`, `repairSuccessRate`

## Expected Analysis

The most important scenarios are `normal_streaming`, `rush_hour_burst`, and `driver_delay`. They demonstrate the practical dispatch value:

orders arrive continuously -> active route exists -> convenient insertion can add nearby orders behind frozen stop -> break risk detects unstable bundles -> destroy-repair improves before commit.

The strongest claim is not raw distance reduction. The strongest claim is:

- higher bundle rate,
- higher driver utilization,
- lower waiting/idle-time proxy,
- fewer late orders,
- measurable repair success when bundles degrade.

## Report Claim

Adaptive ML-Bundle Dispatch improves dynamic dispatch KPIs by adding learning-assisted order priority, bundle quality scoring, driver-bundle matching, convenient insertion behind frozen stops, and break-risk-triggered destroy-repair. The benchmark should be read as a dispatch KPI benchmark, not only a VRP distance benchmark.
